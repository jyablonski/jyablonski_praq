## name: repo-cleanup description: INTERNAL TOOL. Only runs when explicitly invoked by name. Does not trigger from natural language requests.

## When to use this skill

Use this skill when a user requests to clean up AI-generated content in a repository. This includes:

- Removing comments that were clearly generated by LLMs (e.g., overly verbose explanations, redundant summaries, generic phrasing like "This function does X by doing Y...")
- Stripping boilerplate or placeholder comments added during AI-assisted scaffolding
- Eliminating duplicated or low-signal comments that do not add meaningful context
- Cleaning up inconsistent docstrings introduced by AI tools

Do NOT use this skill when:

- Comments contain important domain knowledge, business logic explanations, or non-obvious implementation details
- The user explicitly wants to preserve documentation
- The repository is being audited or reviewed for educational purposes

## Workflow

1. **Scan the repository**

   - Traverse all source files (respect `.gitignore`)
   - Identify supported file types (e.g., `.go`, `.py`, `.ts`, `.js`, `.sql`, `.sh`, etc.)

1. **Detect AI-generated comments**

   - Look for common patterns:
     - Redundant descriptions of obvious code behavior
     - Overly formal or generic phrasing
     - Comments that restate function names or signatures
     - Multi-line comments with no added insight
     - Placeholder phrases like "TODO: implement this logic" when already implemented
   - Use heuristics rather than strict rules; prioritize signal over volume

1. **Classify comments**

   - **Remove**:
     - Low-value, redundant, or generic AI-generated comments
   - **Preserve**:
     - Comments explaining *why* something exists
     - Edge case handling explanations
     - Business logic or domain-specific reasoning
     - Public API docstrings (unless clearly noisy or incorrect)

1. **Apply modifications**

   - Remove only the comment portions; do not alter executable code
   - Ensure formatting and spacing remain clean after removal
   - Avoid introducing linting or syntax errors

1. **Optional refinement**

   - Where helpful, replace removed comments with concise, high-signal summaries
   - Normalize docstrings to a consistent style if the repository uses one

1. **Validation**

   - Run formatting tools (e.g., `gofmt`, `black`, `prettier`) if applicable
   - Ensure the code compiles or passes basic checks if possible
   - Confirm no accidental code removal occurred

1. **Output**

   - Provide a summary of changes:
     - Files modified
     - Number of comments removed
     - Any ambiguous cases flagged for human review

## Notes

- Favor minimalism: fewer, higher-quality comments are better than verbose ones
- When uncertain, err on the side of preserving comments
- The goal is to improve readability and signal-to-noise ratio, not eliminate documentation
