# LLM-as-Judge

## 1. What it solves

LLM outputs are open-ended: many different summaries, support answers, or SQL explanations can be valid. String equality, BLEU, and ROUGE measure surface overlap, not whether the output is good.

| Method | Cost | Latency | Scale | Best use |
| ----------------------- | ------- | ------- | ----- | -------------------------------------- |
| Deterministic assertion | ~0 | ms | High | Narrow, checkable properties |
| Human review | High | Days | Low | Ground truth and high-stakes decisions |
| LLM-as-judge | Low-ish | Seconds | High | Open-ended quality properties |

The basic idea is to use one LLM as a judge for a response generated by another LLM. The judge checks the response against a rubric so we can monitor results, catch quality drift, or, in inline cases, stop known failure paths from being returned to the user.

The judge is a measurement instrument, not a source of truth: it has bias, drift, and a noise floor, so calibrate it against human references before trusting its readings.

Use deterministic checks wherever they exist. Judges are for the residual: genuinely open-ended properties such as groundedness, relevance, tone, completeness, and instruction following.

## 2. How to build one

### Grading modes

- Pointwise: score one output against a rubric. Easy to aggregate, but absolute scores are poorly calibrated and tend to cluster.
- Pairwise: compare a baseline with a candidate and choose A, B, or tie. Relative comparison is usually more reliable and directly answers whether a change helped.
- Reference-based: compare the output with a gold answer or required facts. Best when ground truth exists but surface forms vary.

### Rubrics

- Evaluate one dimension per call. Separate helpfulness, accuracy, tone, and safety instead of producing one smeared composite.
- Prefer binary or three-point verdicts to 1–10 scores.
- Define criteria operationally: “answers the user’s question without requiring a follow-up” is checkable; “is helpful” is not.
- Include examples of passes, failures, and near-misses.
- Ask for reasoning before a structured verdict so disagreements can be audited.

### Contract and configuration

Always require structured output, for example:

```json
{
  "reasoning": "The response cites a section that is not in the source.",
  "verdict": "fail",
  "failure_mode": "unsupported_citation"
}
```

Pin the judge model version, prompt, rubric, and temperature. The judge should generally be stronger than the evaluated model, or from a different model family. Treat prompt and rubric changes like schema changes.

## 3. Measurement discipline

Absolute scores are weak signals. Prefer deltas on a frozen, versioned set: for example, a candidate’s pairwise win rate against a pinned baseline.

Freeze these together:

1. The eval items and references.
1. The judge model.
1. The judge prompt and rubric.

Before interpreting a change, run the same candidate twice to measure the noise floor. Use paired comparisons, bootstrap confidence intervals, and per-dimension results. Report the interval and the dimensions that moved; an overall average can hide a regression.

Keep a held-out set to catch overfitting, refresh production-derived items to catch distribution drift, and re-anchor the judge to human labels after judge changes and on a regular cadence. A small adversarial set of subtly wrong, verbose-but-empty, and correct-but-rude outputs should also be part of calibration.

## 4. Where the judge runs: three placements

These placements are separated by what the verdict does:

| Placement | Verdict feeds | Operating mode |
| --------- | ---------------------------- | -------------------- |
| Inline | An action on this request | Prevention |
| Async | A trend line or triage queue | Monitoring |
| CI | A merge or deployment gate | Regression detection |

Decision rule: action on the request means inline; an aggregate means async; a deploy or merge gate means CI.

Inline and async are not the same production loop. Inline has a request latency budget and low tolerance for false positives. Async can spend more time and tolerate more uncertainty because it feeds aggregates or review queues. CI has the broadest budget and the highest tolerance for false positives because it protects releases.

### Inline

Use a small number of cheap, binary rubrics for failure modes that justify blocking or changing this request. Make the availability policy an explicit `fail_open` or `fail_closed` flag: unless the system fails open, availability becomes generation availability × judge availability. False positives can silently cost revenue without appearing in ordinary error metrics. Cap regeneration or retry attempts so p99 latency cannot grow without bound.

### Async

Score production traces or responses outside the request path and use the results for trends, triage, and human review. The judge can be stronger than the inline judge, but should still be cheaper than a release-gate setup. At low volume, batch nightly rather than streaming and inspect every failure.

### CI

Use the real prompt, model, response, and judge. Mocks test plumbing, not quality. Assert on aggregates, not individual items. Measure the noise floor before choosing a threshold, and gate on regression against a versioned baseline rather than an absolute quality bar. Store the baseline as a versioned artifact updated on merge so drift cannot silently ratchet it downward.

Run CI when something that can change quality changes: prompts, model versions, retrieval configuration, tool-schema descriptions, upstream context, or sampling parameters. Check in this order: deterministic assertions, reference-based checks, groundedness, then open-ended pairwise comparison. Most teams should stop as soon as a trustworthy cheaper check answers the question.

Real-request CI costs more than $0: it pays for generation, retrieval or tool calls, and judge calls. For a critical product generating $100M+ per year, that cost can be worthwhile for confidence that merged changes will not damage a valuable path. For a non-critical quality-of-life feature, a real-request gate is probably not worth the spend; use deterministic checks, a small manually run eval, or lighter CI instead.

## 5. Volume dependence

At low production volume, rates are statistically dead: daily variance hides real shifts and weekly windows detect regressions too late. Replace rate estimation with triage: score everything and read every failure. A short list of concrete bugs is more useful than an unresolvable rate.

- Do not sample at low volume; sampling controls cost at scale.
- Batch nightly instead of streaming.
- Alert on failure kind, not failure rate. Failure kind needs much less volume.
- Low volume shifts weight toward CI because production monitoring reports too late.

At high volume, async sampling can estimate trends and control cost. Keep the sample policy explicit and monitor judge-call failures as missing data, never as passes.

## 6. When all three make sense

Running all three can make sense when a failure is costly in the moment, production behavior can drift, and prompt or model changes need a release gate. They should not be the same judge copied into three places:

- Inline: few binary rubrics, cheap model, strict latency, low false-positive tolerance.
- Async: broader monitoring, between inline and CI in cost and latency.
- CI: many rubrics, strongest model, high false-positive tolerance, aggregate gate.

Sequence the work as traces → CI → async → inline. First observe real failures, then build a regression set and gate changes, then monitor production drift, and only then block requests for failure modes that async has measured. Building inline first tends to protect imagined problems while real regressions remain uncaught.

Share one rubric registry in git, one result schema with a `loop` column, and pinned judge versions across all placements. Otherwise the three systems become unrelated sources of truth.

Good fits for all three include a customer-facing assistant where unsafe answers should be blocked, quality drift should be monitored, and prompt or model changes should be gated; a RAG system where inline checks catch unsupported claims, async checks track retrieval and groundedness, and CI protects against retrieval regressions; and a tool-using agent where inline checks protect tool calls, async checks find workflow failures, and CI tests prompt, model, and tool-schema changes.

## 7. Biases and failure modes

| Bias | Mitigation |
| --------------------------------- | ----------------------------------------------------------------- |
| Position bias in pairwise grading | Run both orderings and count only consistent verdicts as decisive |
| Verbosity or formatting bias | Include concise-but-correct and polished-but-wrong examples |
| Self-preference | Use another model family or an ensemble |
| Sycophancy | Do not leak the expected verdict into the prompt |
| Score compression | Use binary or three-point scales |
| Leniency on subtle errors | Use references and fact-by-fact checks |

Judges are weak at domain-expert correctness, execution, arithmetic, novel safety cases, long-document factuality, and calibrated absolute scores. Run code and SQL, calculate numbers, and require human sign-off where stakes demand it.

## 8. When to implement

Strong fit when outputs are open-ended, prompts or models change frequently, failures are diffuse, or human review cannot keep up. It is a poor fit when deterministic ground truth exists, the dataset is tiny or unrepresentative, human sign-off is mandatory, or nobody will act on the result.

Build the eval asset before building elaborate CI apparatus:

1. Collect real inputs across the failure modes you actually see.
1. Have a domain owner label a calibration sample.
1. Write rubrics and measure judge agreement with those labels.
1. Add a small real-traffic eval and run it manually before shipping.
1. Graduate to tiered, merge-blocking CI only after demonstrated pain and a measured noise floor.
1. Add async monitoring and scoped inline prevention as the evidence requires.

Tiered CI is the mature version, but it is over-built at low volume. Compute is rarely the constraint; the eval set is.

## 9. Cost and storage

- Cache results by `hash(input, output, judge_version, rubric_version)` so unchanged items are not re-evaluated.
- Store each result as a row: `run_id, item_id, model_version, prompt_version, judge_version, rubric_version, dimension, verdict, failure_mode, reasoning, latency, cost, loop`.
- Fan out judge calls within rate limits. Retries and provider failures are expected; keep them distinguishable from passes.

## 10. Quick checklist

- [ ] Real-traffic eval set, human-labeled and stratified
- [ ] Judge model, prompt, rubric, and versions pinned
- [ ] Structured output with a failure kind
- [ ] Binary or three-point rubrics, one dimension per call
- [ ] Noise floor and confidence intervals measured
- [ ] Per-dimension reporting and held-out data
- [ ] Human agreement and adversarial calibration checked
- [ ] Deterministic checks used wherever possible
- [ ] Placement chosen deliberately: inline, async, CI, or a justified combination
- [ ] Results and baselines versioned for forensics

The durable discipline is simple: freeze the instrument, measure its noise, calibrate it against humans, and report changes rather than pretending an absolute score is truth.

## 11. Example: a 25-model production system

Imagine a large customer-support and search company running roughly 25 specialized LLMs in production: routers, query rewriters, retrieval and ranking helpers, answer generators, safety checks, summarizers, translators, and tool-use agents. The models may run in sequence or in parallel, but that does not mean every intermediate response needs its own judge. Judge the user-visible boundary and the places where changes can create meaningful risk.

Good candidates:

- Inline: check that a customer-facing answer is grounded, follows policy, and is safe to return; or check a proposed tool action before execution.
- Async: score production answers after the fact to find drift by model, route, customer segment, or failure kind, and send suspicious cases to human review.
- CI: replay a set of real traces when a prompt, model, retrieval setting, or tool description changes, then gate on aggregate regressions in answer quality or tool-use success.

Poor candidates:

- JSON schema, permissions, tool-argument validation, HTTP behavior, and other properties that deterministic assertions can check more cheaply and reliably.
- Code or SQL correctness, where tests or executing the query provide stronger evidence than a judge.
- Every intermediate model response when there is no observed failure mode; the extra latency and cost are not justified by a hypothetical problem.
- A low-stakes quality-of-life feature with little traffic or business impact; use a small manual eval or lightweight checks instead.
